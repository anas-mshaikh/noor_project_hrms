# DMS + Employee Documents v1 (Milestone 5)

This repo includes an enterprise-safe **DMS (Document Management System)** layer under schema `dms`.

Design principles:
- `dms.files` stores canonical file metadata; bytes live on disk (LOCAL provider in v1).
- `dms.documents` stores canonical business records (employee-owned documents) and supports verification lifecycle.
- `dms.document_versions` is immutable history; `dms.documents.current_version_id` points at the active version.
- Workflow Engine v1 drives approvals/verification; terminal transitions trigger domain hooks.
- Expiry alerts are generated by a worker into `dms.expiry_events` and delivered via in-app notifications.

## Database

Core tables (schema `dms`):
- `dms.files`: file metadata + storage reference (`object_key` is a relative path under `backend/data/`)
- `dms.document_types`: tenant-scoped catalog (code, name, requires_expiry, is_active)
- `dms.documents`: employee-owned record (status: `DRAFT|SUBMITTED|VERIFIED|REJECTED|EXPIRED`)
- `dms.document_versions`: immutable versions (unique by tenant/document/version)
- `dms.document_links`: polymorphic links to attach documents to domain entities
- `dms.expiry_rules`: rules per document type (days_before, is_active)
- `dms.expiry_events`: materialized events (notify_on_date, event_type) used by the expiry worker

## Local storage (v1)

Files are stored under `backend/data/dms/` and referenced by `dms.files.object_key`.

Canonical path pattern:
- `dms/files/{tenant_id}/{file_id}/{safe_filename}`

## Workflow integration (document verification)

Workflow request type:
- `DOCUMENT_VERIFICATION` (seeded via Alembic)

When a verification workflow reaches a terminal status, the workflow engine invokes a domain hook:
- `entity_type="dms.document"`
- `entity_id=dms.documents.id`

Hook side-effects:
- Approved: set `dms.documents.status='VERIFIED'`, set `verified_at/verified_by_user_id`, enqueue in-app notification to the document owner.
- Rejected: set `status='REJECTED'`, set `rejected_reason`, enqueue in-app notification to the owner.
- Cancelled: clears `verification_workflow_request_id` to allow resubmission.

## Expiry alerts (worker)

Worker module:
- `backend/app/worker/dms_expiry_worker.py`

Behavior:
- Materializes EXPIRING events for a lookahead window into `dms.expiry_events` idempotently.
- Emits due notifications into `workflow.notification_outbox` using a stable `dedupe_key`.
- Tests invoke `run_once()` directly; production can run it daily via cron/K8s.

## Permissions (colon-style)

Seeded and mapped via Alembic:
- `dms:file:read`, `dms:file:write`
- `dms:document-type:read`, `dms:document-type:write`
- `dms:document:read`, `dms:document:write`
- `dms:document:verify`
- `dms:expiry:read`, `dms:expiry:write`

Important access control rules:
- File reads are tenant-scoped and "reachable" (owner document version OR workflow participant), unless the actor has HR/admin DMS permissions.
- ESS document reads are participant-only: employees can only see their own documents (404 on deny).

## API (all under `/api/v1`)

Files:
- `POST /dms/files` (multipart: file)
- `GET  /dms/files/{file_id}` (metadata; enveloped)
- `GET  /dms/files/{file_id}/download` (bytes)

Document types:
- `GET   /dms/document-types`
- `POST  /dms/document-types`
- `PATCH /dms/document-types/{id}`

Employee document library:
- HR:
  - `POST /hr/employees/{employee_id}/documents`
  - `GET  /hr/employees/{employee_id}/documents?status=&type=&limit=&cursor=`
- ESS:
  - `GET /ess/me/documents?status=&type=&limit=&cursor=`
  - `GET /ess/me/documents/{document_id}`

Versioning:
- `POST /dms/documents/{document_id}/versions`

Verification (workflow):
- `POST /dms/documents/{document_id}/verify-request`

Expiry:
- `GET  /dms/expiry/rules`
- `POST /dms/expiry/rules`
- `GET  /dms/expiry/upcoming?days=30`
